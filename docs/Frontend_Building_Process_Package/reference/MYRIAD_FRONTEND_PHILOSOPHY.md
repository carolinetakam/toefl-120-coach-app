# Myriad Frontend Philosophy

This file captures the frontend philosophy demonstrated in Myriad and generalizes it for Caroline's apps.

## Product Workspace First

Every app should open directly into the useful product experience.

Avoid:

- oversized hero sections
- marketing-first layouts
- decorative dashboards
- placeholder panels
- long in-app explanations of how the app works

Prefer:

- real workflow surface
- active context
- visible state
- next action
- concise controls

## Calm, Dense, Premium

The visual language should feel like a serious operating workspace.

Use:

- light warm neutral surfaces
- soft panel depth
- compact information hierarchy
- restrained spacing
- tactile controls
- clear status indicators
- one accent color
- polished empty/loading/error states

Avoid:

- generic admin-template styling
- visual clutter
- decorative blobs/orbs
- excessive gradients
- one-note color palettes
- fake complexity

## Human Dignity And Agency

The app should help people become clearer, calmer, and more capable.

Technology is the servant, not the master.

The frontend should not exploit:

- fear
- insecurity
- vanity
- outrage
- compulsive desire
- artificial urgency
- endless attention capture

Good UI should provide:

- clarity
- meaningful choice
- calm pacing
- visible exits
- understandable automation
- honest status
- user-controlled authority

## Beauty Serves Clarity

Beauty is not decoration.

In these apps, beauty should:

- communicate care
- reduce friction
- create trust
- make state readable
- make workflows feel safe and manageable
- help the user focus

If a visual decision does not improve clarity, trust, or workflow quality, cut it.

## Tactile, Grounded Interface

The preferred feel is:

- paper-like
- editorial
- tactile
- lightly dimensional
- mechanically calm
- physically believable

Use soft raised panels, inset wells, compact controls, and grounded spacing.

Do not make the interface feel weightless, overstimulating, or artificial.

## Myriad Pattern

Myriad's reusable frontend pattern:

- product loop is visible
- app shell holds context
- data status is truthful
- agent/AI actions are observable
- approval authority is clear
- Brand Brain or product memory is visible
- workflow cards show source, status, owner, risk, and next action
- results tie back to decisions, not vanity metrics

For another app, replace Myriad's loop with that app's core loop.

Examples:

- TOEFL app: profile -> diagnostic -> review -> next drill -> progress
- business app: research -> offer -> launch -> sales -> metrics
- creative app: idea -> draft -> produce -> publish -> learn
- operations app: intake -> queue -> work -> review -> deliver
- AI app: input -> agent work -> review -> approve -> audit

## Language Standard

Use concise operational language.

Good:

- `Ready to review`
- `Saved locally`
- `Live data`
- `Needs approval`
- `Updated 2 min ago`
- `3 items blocked`

Avoid:

- vague motivational copy
- fake urgency
- long instructional paragraphs
- claims the system cannot prove
